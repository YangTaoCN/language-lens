# Language Lens - 需求文档 (Requirements Document)

## 1. 项目概述 (Project Overview)
本项目旨在开发一款基于 Web 的语言学习应用 "Language Lens"，利用图像识别和 OCR 技术，将日常生活中的图片转化为交互式学习材料。

## 2. 核心功能 (Core Features)

### 2.1 语言设置 (Language Settings)
*   **学习语言 (Source Language):** 用户希望学习的语言（例如：日语、西班牙语、法语等）。
*   **母语/目的语言 (Your Language):** 用户熟悉的语言，用于查看翻译解释（例如：中文、英文）。
*   **持久化存储:** 系统应保存用户的语言偏好。

### 2.2 图片处理与 OCR (Image Processing & OCR)
*   **图片输入:**
    *   **拍照:** 调用设备摄像头实时拍摄。
    *   **上传:** 从本地相册或文件系统上传图片。
*   **文字识别:** 系统自动识别图片中的“学习语言”文字。
*   **区域定位:** 记录每个识别文本块（单词或短语）在图片中的坐标位置。

### 2.3 交互式学习 (Interactive Learning)
*   **热区生成:** 在原始图片上覆盖透明的可点击区域（热区），对应识别出的文字位置。
*   **智能提示 (Smart Tooltip):** 点击热区后，弹出详细信息框，包含：
    *   **原文:** 识别到的单词/短语。
    *   **发音:** 提供该单词在“学习语言”中的发音（TTS 播放或音标）。
    *   **释义:** 该单词在“目的语言”中的翻译和解释。
    *   **视觉辅助:** 提供 Google 图片搜索链接，帮助用户建立图像联想。

## 3. 技术栈建议 (Tech Stack)
*   **前端:** Next.js (React), Tailwind CSS
*   **后端:** Next.js API Routes (Serverless Functions)
*   **OCR 服务:** Google Cloud Vision API 或 Tesseract.js (离线/轻量级方案)
*   **翻译/TTS:** Google Translate API, Azure Cognitive Services, 或 OpenAI API

## 4. 后续规划 (Roadmap)
*   **生词本:** 保存难点单词。
*   **历史记录:** 查看过往上传的图片。
*   **多语言界面:** 支持不同语言的 UI。

## 5. UI 需求 (UI Requirements)

为提高学习体验并解决当前原型中的可用性问题，界面应遵循以下设计与交互要求：

- **整体视觉与信息层次**：采用清晰、现代的学习应用配色（主色用于交互控件，辅助色用于次要信息），保证文字可读性与足够的对比度；在主视图中突出图片与交互提示，侧边栏提供设置/生词本/历史等次要面板。

- **可响应布局**：在桌面、平板、手机上都应有良好体验。大屏时采用左右两栏（左侧控制/生词本，右侧图片与热区）；小屏上采用堆叠布局并通过抽屉或切换按钮访问生词本与设置。

- **图片查看区域（可缩放/平移）**：图片区域必须支持放大/缩小和平移（鼠标滚轮 + 拖拽、触控板缩放、移动端捏合/双击放大），以便查看小字。提供可见的缩放控件（+ / - / 适合屏幕）以及重置视图按钮；默认适配容器宽度并允许放大至少到 200%-400% 以识别小字体。

- **热区（非侵入式）**：热区不应以醒目黄色边框恒显，避免影响阅读。
  - 默认情况下热区为透明（不可见），仅在 hover / focus / 长按 或在“显示热区”开关打开时，显示微妙的高亮（例如淡色半透明背景或柔和描边），并提供轻微动画反馈。
  - 热区应在放大/缩放/平移后保持对齐（bounding box 基于原始图像像素坐标，实时转换为当前视图坐标）。

- **智能提示（Tooltip / Card）**：点击热区显示卡片，包含：原文、翻译、发音按钮（TTS 播放）、音标（如果可用）、Google 图片链接、收藏/加入生词本按钮，以及相关示例或词义扩展。卡片应使用 shadow + rounded 样式，且不会遮挡主要阅读区域（可根据点击位置自动选择卡片方向）。

- **加载与占位**：当 OCR 或翻译正在运行时，图片区域和卡片应显示轻量级 loading 指示（spinner 或骨架屏），避免布局跳动。

- **生词本与历史面板**：生词列表应可排序/筛选（时间、字母、难度）；每条记录展示原词、翻译、添加时间、发音按钮与“跳转回图片”功能（如果可用）。支持导出/导入（JSON）备份。

- **无障碍与键盘导航**：所有交互元素（热区、卡片按钮、缩放控件、生词项）应可通过键盘导航（Tab / Enter / Space）访问，并提供 aria-label、role 与可读性良好的焦点样式。支持屏幕阅读器的文本替代与显式语言声明。

- **错误与提示信息**：对 OCR、翻译、网络失败等场景提供友好错误提示与重试操作（例如："识别失败，重试"）。对于权限问题（摄像头拒绝）提供明确说明与手动上传替代方案。

- **设置与可定制性**：允许用户开关热区可见性、切换热区高亮样式（悬浮/描边/仅提示）、调整默认缩放级别与 TTS 语速/音量（基于浏览器或后端 TTS）。

- **性能优化**：对大图进行按需缩放/瓦片化或在前端使用 canvas 渲染以降低内存与重排开销；OCR 请求应去抖动（debounce）并限制并发；热区渲染采用批量化更新以避免大量 DOM 节点引起的卡顿。

- **视觉细节与动效**：使用细腻的过渡（120-200ms）和轻微的阴影来增强触控反馈，避免使用过度鲜艳或大面积的高饱和色块；按钮与控件保持一致的圆角与间距以提升亲和力。

- **移动端交互**：移动端支持单手可达的控件摆放（底部操作栏或悬浮行动按钮），支持长按显示卡片与拖动图片平移，避免在移动端强制显示大面积热区遮挡内容。

以上 UI 需求既作为产品设计方向，也作为下一步实现细节的验收标准。后续实现时应逐条对照这些要点进行开发和测试。
